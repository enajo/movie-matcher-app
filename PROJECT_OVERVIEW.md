## 1. One-paragraph summary

Movie Matcher is a hybrid movie recommender system with a Flask backend and a
single-page HTML/JS frontend. It combines two independent recommendation strategies:
(1) a **content-based filtering** engine that recommends movies similar to a chosen
title, based purely on metadata (plot, genre, keywords, cast, director), computed
offline with TF-IDF + PCA + cosine similarity; and (2) a **collaborative filtering**
engine using **matrix factorization (SVD)** that learns personalized taste from a
user's own thumbs-up/thumbs-down ratings compared against everyone else's ratings,
trained live with `scikit-surprise`. The two are not blended — they power two
separate features in the UI ("Similar to X" vs. "For You").

## 2. Dataset

**Source:** TMDB 5000 Movie Dataset (a well-known public Kaggle dataset derived from
The Movie Database).

| File | Rows | Content |
|---|---|---|
| `tmdb_5000_movies.csv` | ~4,800 | `title`, `overview` (plot summary), `genres` (JSON), `keywords` (JSON), `id` |
| `tmdb_5000_credits.csv` | ~4,800 | `movie_id`, `cast` (JSON list of actors), `crew` (JSON list incl. director) |

The two files are merged on `movie_id` / `id`. This is the **catalog** — static,
loaded once, never changes at runtime.

**Second data source — live user ratings** (`data/ratings.json`), generated by the
app itself as people use it:

```json
{"user_id": "abc123", "title": "Inception", "rating": 1, "ts": 1751500000}
```

- `rating` is binary: `1` (thumbs up) or `-1`/`0` (thumbs down) — a simplified
  implicit/explicit rating scale (−1 to 1), not a 5-star scale.
- Re-rating the same title by the same user overwrites the previous entry.
- This file is the training data for the collaborative-filtering model — it starts
  empty and grows only from real usage, which is why that feature has a cold-start
  gate (see §4.2).

## 3. Motivation scenario (for the paper's §3)

Suggested framing, grounded in what the system actually does — adapt freely:

> **Problem:** A user facing a catalog of thousands of movies suffers from choice
> overload. Two distinct needs exist simultaneously: (a) "I just watched/liked movie X,
> what else is like it?" — a need best served by *content similarity*, and (b)
> "given everything I've rated so far, what would *I specifically* enjoy?" — a need
> best served by *collaborative filtering*, which captures taste patterns that pure
> content matching cannot (e.g., users who love both quiet dramas and violent action
> films for orthogonal reasons content metadata won't reveal).
>
> **Why a hybrid, not just one:** content-based recommendations work from the first
> click (no history needed) but tend to over-recommend near-duplicates and can't
> learn cross-genre taste. Collaborative filtering captures genuine taste signal but
> is useless for a brand-new user or a brand-new item (cold-start problem). Movie
> Matcher deliberately keeps both, gated by data availability, rather than
> prematurely blending them into one score.
>
> **Healthcare-scenario alternative** (if the paper is steered that direction instead):
> the same architecture maps directly onto a *clinical decision-support or patient
> education recommender* — e.g., recommending health articles/interventions similar
> to one a clinician is viewing (content-based, metadata = condition/symptom/keyword
> tags) versus recommending interventions based on patterns across similar patients'
> recorded outcomes (collaborative filtering / matrix factorization on
> patient-treatment-outcome triples instead of user-movie-rating triples). The
> cold-start problem is identical: a new patient has no history, so content-based
> similarity must carry the recommendation until enough outcome data exists.

### 4.2 Algorithm 1 — Content-based filtering

**Goal:** given a movie title, return the N most similar movies.

**Offline pipeline** (`preprocessing.py`):
1. Merge movies + credits on ID.
2. Drop rows missing `overview`, `genres`, `keywords`, `cast`, or `crew`.
3. Parse the JSON-encoded `genres`, `keywords`, `cast`, `crew` columns
   (`ast.literal_eval`), extract: all genre names, all keyword names, top-3 cast
   names, the director's name (first crew entry with `job == "Director"`).
4. Normalize each list to lowercase, no-space tokens (e.g., `"Tom Hanks"` →
   `"tomhanks"`) so multi-word names count as a single token.
5. Concatenate `overview` tokens + genre tokens + keyword tokens + cast tokens +
   director token into one space-joined string per movie — the **"tags" metadata
   soup**.
6. **TF-IDF vectorization**: `TfidfVectorizer(max_features=5000, stop_words="english")`
   turns each movie's tag string into a 5000-dimensional sparse vector, weighting
   words that are frequent *in that movie* but rare *across the whole catalog*
   higher (down-weighting generic words like "action" that appear everywhere,
   up-weighting distinctive words like a specific character or actor name).
7. **PCA dimensionality reduction**: 5000 → 300 dimensions, to reduce noise and
   computation cost while retaining most of the variance (percentage logged at
   build time).
8. **Cosine similarity**: pairwise cosine similarity across all 300-dim vectors
   produces a dense 4800×4800 matrix — `similarity[i][j]` = how similar movie *i*
   is to movie *j*.
9. Persist `movies_processed.pkl` (cleaned catalog) and `similarity.pkl` (the
   matrix, ~184 MB) to disk.

**Runtime lookup** (`recommender.py`, `recommend_movies()`):
- Find the row index of the requested title.
- Read that row of the precomputed similarity matrix.
- Sort by score descending, skip the movie itself (index 0 after sort), return the
  top N with title, genre string, poster URL, and similarity score.
- **No machine learning happens at request time** — it's array indexing and a sort.
  All the expensive computation is front-loaded into the offline build step.

**Why this works without any user data:** it's a pure item-item similarity system;
new users get useful recommendations immediately (no cold-start problem for the
*user* side — though a brand-new *movie* added to the catalog would need the whole
pipeline rerun, which is a cold-start problem for the *item* side).

### 4.3 Algorithm 2 — Collaborative filtering via matrix factorization (SVD)

**Goal:** given a specific user's rating history, predict how much they'd like
every movie they haven't rated yet, personalized to their taste rather than to
metadata.

**Library:** `scikit-surprise`, using its `SVD` class — a **Funk-SVD / latent
factor model** (the algorithm popularized in the Netflix Prize competition), not
literal linear-algebra Singular Value Decomposition. True SVD requires a fully
dense matrix; the user-movie ratings matrix here is extremely sparse (most users
haven't rated most movies), so Funk-SVD instead *learns* the factorization via
gradient descent.

**Mechanics** (`utils/svd_model.py`):
1. Build a `(user_id, title, rating)` DataFrame from all ratings collected so far.
2. `Reader(rating_scale=(-1, 1))` tells Surprise the rating range.
3. `SVD(n_factors=50, n_epochs=25, lr_all=0.005, reg_all=0.02)`:
   - **`n_factors=50`**: each user and each movie is represented as a 50-dimensional
     latent vector — hidden "taste dimensions" learned automatically (not
     hand-labeled genres; the model finds its own structure, e.g. it might
     implicitly learn something like "prefers slow character-driven films" as a
     direction in that 50-dim space).
   - **`n_epochs=25`**: 25 passes of gradient-descent training over the ratings.
   - **`lr_all` / `reg_all`**: learning rate and L2 regularization strength, to
     avoid overfitting to a small ratings set.
4. A predicted rating for (user, movie) = **dot product of the user's latent
   vector and the movie's latent vector** (plus learned bias terms), trained to
   minimize error against the *known* ratings.
5. **Training gate:** the model only trains at all once **≥5 ratings exist in
   total**, across all users (`train()` in `svd_model.py` returns `None` below
   that).
6. **Caching / retraining policy:** the trained model is cached in memory
   (`_model` global) and is only retrained when the *total ratings count changes*
   — a simple dirty-check, no scheduled retraining job.
7. **Per-user gate:** even once a global model exists, an individual user only
   sees personalized results once **they personally have rated ≥10 movies**
   (`server.py`, `/api/recommend/me`) — this is a separate, stricter threshold
   than the model-training minimum, meant to ensure the model actually has enough
   signal about *that specific user* to be meaningfully personalized, not just
   technically able to produce a number.
8. **Inference** (`predict_for_user()`): for every movie the user hasn't rated,
   call `model.predict(user_id, title)`, sort by predicted score, return the top 12.

### 4.4 Data flow / request flow

| Step | Trigger | Component | Result |
|---|---|---|---|
| 1 | Build time (once, offline) | `preprocessing.py` | `movies_processed.pkl`, `similarity.pkl` |
| 2 | Server boot | `server.py` | loads both pickles into memory |
| 3 | User browses/searches | `GET /api/movies` | filtered movie list + posters (TMDB API, cached in-process) |
| 4 | User picks a movie | `GET /api/recommend?title=X` | content-based top-12 similar movies |
| 5 | User rates a movie | `POST /api/rate` | appended to `ratings.json` |
| 6 | User has rated ≥10 | `GET /api/recommend/me?user_id=Y` | SVD model (re)trained if needed, top-12 personalized picks |
| 7 | UI display | `GET /api/ratings`, `/api/ratings/me` | aggregate 👍/👎 counts per title, per-user rating status |

### 4.5 Tech stack

- **Backend:** Python 3.9, Flask 3.1, `scikit-learn` (TF-IDF, PCA, cosine similarity),
  `scikit-surprise` (SVD/matrix factorization), `pandas`, `numpy`, `python-dotenv`.
- **Frontend:** server-rendered single HTML template (`templates/index.html`,
  ~1,370 lines) with vanilla JavaScript calling the JSON API via `fetch` — no
  frontend framework.
- **External API:** TMDB (`api.themoviedb.org`) for poster images, called
  server-side with a bearer token, results cached in-process (`_poster_cache`) and
  batch-fetched concurrently (`ThreadPoolExecutor`) to avoid one-request-per-poster
  latency.
- **Persistence:** flat files — pickled matrices for the catalog/similarity data,
  a JSON file for ratings. No database. Adequate for a single-instance prototype;
  would not scale to concurrent writers or a larger catalog without migrating
  ratings storage to a real DB.
- **Legacy artifact:** `app.py` is an earlier Streamlit-based prototype UI, superseded
  by the current Flask + HTML frontend but still present in the repo.

## 5. Mock-up / UI design (for the paper's §5)

The interface is a single-page dark-themed web app (`templates/index.html`).
Key regions, inferred from the template's structure:

- **Navbar** (`navbar`, `nav-brand`, `nav-logo-box`): app branding, plus a user
  avatar/name area (`nav-avatar`, `nav-user-name`) representing the session's
  anonymous `user_id`.
- **Hero section** (`hero`, `hero-h1`, `hero-sub`, `hero-ctas`): landing banner
  with primary call-to-action buttons (`btn-primary`, `btn-ghost`).
- **Genre filter strip** (`genre-strip`): horizontal list of clickable genre
  chips, backed by `GET /api/genres`, filters the browse grid.
- **Carousel / coverflow** (`carousel-section`, `coverflow`, `cf-track`):
  a horizontally-scrolling featured-movies strip.
- **Browse grid** (`browse-section`, `browse-heading`, `movie-grid`): grid of
  movie cards (`m-card`) each showing poster (`m-poster`, with `m-poster-empty`
  fallback when no poster is available), title (`m-title`), genre tag
  (`m-genre`), and rating actions (`m-actions` — the thumbs up/down buttons that
  call `POST /api/rate`).
- **Content tabs** (`content-tabs`): likely toggles between "Browse/Similar" and
  "For You" views.
- **"For You" personalized section** (`fy-header`, `fy-title`, `fy-subtitle`,
  `fy-badge`, `fy-refresh-btn`): the SVD-powered personalized recommendation
  panel, with a refresh action to re-pull `/api/recommend/me` and a badge
  (likely showing SVD readiness/score).
- **Progress indicator** (`progress-wrap`, `progress-bar-bg`, `progress-bar-fill`,
  `progress-label`): visualizes how close the current user is to the 10-rating
  threshold needed to unlock personalized recommendations — directly surfaces the
  cold-start gate from §4.3 to the user as an onboarding incentive ("rate N more
  movies to unlock").
- **Debug panel** (`debug-panel`, `debug-header`, `debug-session`,
  `debug-rated-list`, `debug-rated-item`): a developer-facing overlay listing the
  current session's `user_id` and every rating cast — useful for demoing/testing
  the cold-start gate without needing 10 real ratings typed out by hand.

## 7. Known limitations (useful for Discussion/SWOT, §7)

- **Cold-start on the item side:** adding a new movie requires rerunning the
  entire offline pipeline (`preprocessing.py`) to regenerate the 184MB similarity
  matrix — there's no incremental update path.
- **No persistent database:** ratings are stored in a single JSON file with no
  locking; concurrent writes from multiple users at the exact same moment could
  race. Fine for a prototype/demo, not production-grade.
- **Binary/simplified ratings:** the −1/1 rating scale is coarser than a 5-star or
  continuous scale, which limits how much nuance the SVD model can learn from
  each data point.
- **No blending of the two algorithms:** a user who has unlocked personalized
  recommendations still gets *unpersonalized* content-based results on the
  "Similar to X" screen — the two systems are additive, not fused into a single
  ranked list. This is a deliberate simplicity trade-off worth calling out
  explicitly as a "future work" direction (e.g., a weighted hybrid score).
- **Single-process, in-memory poster cache and SVD model:** neither survives a
  server restart or scales horizontally without shared external state (e.g.
  Redis) — worth mentioning if the paper's Discussion touches on real-market
  deployment.
